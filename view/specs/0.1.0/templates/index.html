Templates draft specification 0.1.0

The Problem

Templates are currently just a placeholder, proof of concept. They are severely limited. A default.html must always be defined locally, so even within what is currently the only pipeline 'ViewDrivenPipeline' we have no flexibility.

The Solution

The most flexible possibility is for users to specify a particular template in the head 'furniture'. An example is given below. 

  <html>
    <head>
      <meta name="template" content="template.html">
    </head>
    ...
  </html>

If the specified template is not found Silk will check for the same template in Silk core. If the same template is not found in Silk core a 'missing-template' template is wrapped around the view in question, this template makes it clear to the user that they have not specified an existing template. This information is also displayed by the CLI when Silk spinning occurs.

If no external template is specified then Silk will use 'default.html' in the templates directory.

If for some reason 'default.html' does not exist then Silk will use a 'template-missing' template defined in Silk core.

Silk will be updated with new commands for cloning and installing templates to aid with developing new ones.

Post 0.1.0

Note we have not addressed specifying template structure directly in the view in the head 'furniture', that will come post 0.1.0 and will be implemented at the same time as functionality to enable re-writing elements of the template used by a view (like the title).
